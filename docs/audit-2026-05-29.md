# Cincy Voices Audit — May 29, 2026

## Executive Summary

The pipeline works. The site looks good. The editorial quality is inconsistent. The root cause is exactly what the brief hypothesized: the first pass optimized for transcript text and ignored the actual video moment. Claude picked moments that sounded good on paper, not moments that look and sound compelling on screen. The Gemini test confirmed this with a concrete example: one of the "soundbite" clips in the library is a greeting outtake ("Hey Adam, how you doing? Get out of my podcast") with zero content value. Transcript scoring would never catch that.

The good news: the production quality for leaders who were filmed on-set (wooden slatted wall, boom mic, warm lighting) is consistently strong. The content is genuinely good. The issue is editorial selection, not raw material. A more rigorous recut could make this a substantially better library with the footage that already exists.

### Pipeline Architecture

```
Raw Video
  → transcribe.sh (AssemblyAI: utterances + word timestamps)
  → enrich.sh (Claude: segment enrichment, topic tagging)
  → extract-quotes.sh (Claude: quote extraction)
  → identify-clips.py (Claude: 8 clip candidates per leader from transcript only)
  → cut-clip.sh (ffmpeg: trim master to clip)
  → Mux upload → playback_id stored in Supabase
  → seed-leaders.sh / seed-sources.sh → Supabase leader/source records
  → Site reads Supabase → Next.js display
```

**Critical gap:** the pipeline has no step that evaluates the actual video. Claude sees transcript text and timestamps. It never sees the speaker's face, expression, framing, audio quality, or whether the moment is visually complete.

### Schema Fields Present vs. Missing

**Present in Supabase (`cincy_voices_segments`):**
- id, source_id, leader_id, text, start_time_ms, end_time_ms
- mux_playback_id (stores master playback ID; clip uses trim times)
- clip_quality_score, clip_candidate, review_status
- words (word-level timestamps from AssemblyAI)
- trim_start_ms, trim_end_ms
- topic (string — but not a foreign key)
- topic_thread_id (appears in schema but nullable, seems underused)

**Missing — critical for a better pipeline:**
- `visual_tags` (framing, expression, eye contact, gesture)
- `emotional_tone` (analytical, personal, passionate, vulnerable, urgent)
- `buyer_persona_relevance` (SMB owner, fractional leader, COO, CFO, HR, Sales)
- `function_relevance` (CFO, CMO, COO, HR, Sales, EOS)
- `content_use_cases[]` (public_site, vault, linkedin, broll, sales_deck, roebling_demo)
- `broll_quality_score` (separate from clip_quality_score)
- `audio_quality_score` (no signal on mic quality or background noise)
- `visual_quality_score` (no signal on framing, lighting)
- `rights_consent_status` — **critical gap, nothing tracks consent**
- `suggested_linkedin_copy`
- `clean_quote` (currently `text` field mixes editorial text and transcript)
- `raw_source_playback_id` (separate from clip playback ID — currently confusing)
- `gemini_analysis` (JSON blob from video-understanding pass)

## Phase 4: Proposed New Schema — "Moments" Table

The current schema has `cincy_voices_segments` which conflates source segments (long blocks from the transcript) with clip candidates (short, curated moments). The new approach should separate these.

### New table: `cincy_voices_moments`

```sql
CREATE TABLE cincy_voices_moments (
  id                        UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  
  -- Core identity
  leader_id                 UUID REFERENCES cincy_voices_leaders(id),
  source_id                 UUID REFERENCES cincy_voices_sources(id),
  
  -- Timing
  start_ms                  INTEGER NOT NULL,
  end_ms                    INTEGER NOT NULL,
  duration_ms               INTEGER GENERATED ALWAYS AS (end_ms - start_ms) STORED,
  
  -- Content
  transcript_verbatim       TEXT,           -- Raw AssemblyAI text, filler words included
  transcript_clean          TEXT,           -- Editorial version, cleaned
  pull_quote                TEXT,           -- Shareable one-liner, max 160 chars
  
  -- Classification
  topic_tags                TEXT[],         -- e.g. ['trust', 'culture', 'delegation']
  visual_tags               TEXT[],         -- e.g. ['direct-to-camera', 'animated-hands', 'eye-contact']
```

### Migration strategy

Keep `cincy_voices_segments` for raw transcript segments (the enrichment step output). Add `cincy_voices_moments` as the curated layer above it. A segment can spawn zero or many moments. A moment always references back to its source segment via `source_id + start_ms + end_ms`.

This way, the pipeline becomes:

```
AssemblyAI → segments (raw)
  → Gemini + Claude → moments (curated)
    → Ford review → approved moments
      → Site + vault
```

### What to Change in the Pipeline

1. **Add a Gemini pass after `identify-clips.py`.**  Run every candidate through `gemini-2.5-flash` video analysis before inserting into Supabase. Score `standalone_clarity`, `filler_density`, `clean_start` (does the clip start at a natural sentence boundary?), `clean_end`. Reject any candidate scoring below 5 on standalone_clarity.

2. **Fix the transcript cleaning bug.** The current pipeline asks Claude to "clean the text editorially" in the same step where it extracts clips. This masks filler words and makes scores look better than reality. Keep a `transcript_verbatim` field that preserves the raw text alongside the cleaned version.

3. **Add clip length enforcement.** 10-25 seconds is the spec. The current candidates include 103-second selections. Add a hard reject for anything over 30 seconds. The `identify-clips.py` prompt says "10-25 seconds" but Claude frequently violates this when the content is good — Gemini's duration signal should be used to enforce it.

4. **Add timestamp validation.** There is at least one segment with `start_ms > end_ms` (Amy Sheehy clip C). Add an assertion in `identify-clips.py` and `cut-clip.sh`.

5. **Add consent tracking.** There is currently no field that records whether a leader consented to each clip being published. This is a legal risk. Add `consent_status` to every moment before publishing.

6. **Separate `mux_clip_playback_id` from `mux_source_playback_id`.** The current schema stores the master playback ID in `mux_playback_id` and uses `trim_start_ms`/`trim_end_ms` to define the clip window. This is confusing — the site appears to be serving the full video with trim offsets rather than independently cut clips. Clarify and enforce this distinction.

7. **Add leader profile context to the identify-clips prompt.** The `identify-clips.py` prompt doesn't load the leader's markdown profile. It only has the transcript. The rich context in `/content/leaders/*.md` (origin story, philosophy, power quotes, profile themes) should inform what Claude picks as clip candidates.

### What to Change in the Website

1. **Leader hero clip should be manually selected by Ford.** The current system auto-promotes the highest scored clip. After this recut, Ford should select the hero clip per leader. Use the `hero_segment_id` field in the leaders table — it exists but it's not clear it's being populated.

2. **Add clip length display.** Users can't tell if a clip is 12 seconds or 45 seconds before clicking play. Show duration on all clip cards.

3. **Add "Why this matters" text per clip.** The `pull_quote` and `why_it_matters` fields from the proposed new schema should display on the vault clip card — similar to what a podcast platform does with chapter descriptions.

4. **Reduce public site clips per leader to 3 maximum.** Currently fetches top 5. 3 strong clips beats 5 adequate clips. Save the rest for vault.

5. **Topic pages need buyer framing.** The current topics (trust, delegation, culture, etc.) are labeled from the speaker's perspective. Reframe them for the buyer: "Building a Team You Can Step Away From," "Why Your Best Employee Is Underused," etc.

6. **Add a LinkedIn copy generator to the vault.** Each vault clip should have a "Copy LinkedIn caption" button that surfaces the `suggested_linkedin_copy` field. This turns the vault from passive storage to an active content tool.
